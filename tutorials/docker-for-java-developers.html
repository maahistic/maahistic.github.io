1. Why Docker?

Every Java developer has heard it: "It works on my machine." Docker solves this by packaging your application with everything it needs — JRE, dependencies, configuration — into a portable container.

For Java developers specifically, Docker gives you:

  • Consistent environments — same container runs on dev, staging, and production
  • Easy dependency management — databases, message queues, caches all run alongside your app
  • Fast onboarding — new team members run docker compose up and they're ready
  • Cloud-native readiness — containers deploy to Kubernetes, AWS ECS, Azure AKS seamlessly

3. Your First Dockerfile

Create a Dockerfile in your project root. The simplest approach copies the pre-built JAR:

Dockerfile# Use Eclipse Temurin (free, production-ready JDK)
FROM eclipse-temurin:17-jre

# Set working directory
WORKDIR /app

# Copy the JAR file
COPY target/task-api-0.0.1-SNAPSHOT.jar app.jar

# Expose the port Spring Boot runs on
EXPOSE 8080

# Run the application
ENTRYPOINT ["java", "-jar", "app.jar"]

Build and run it:

Terminal# Build the JAR first
./mvnw clean package -DskipTests

# Build the Docker image
docker build -t task-api:latest .

# Run the container
docker run -p 8080:8080 task-api:latest

Your API is now running inside a container at http://localhost:8080.

Note

We use eclipse-temurin:17-jre (not JDK) because the JRE is smaller (~130MB vs ~300MB) and production apps don't need the compiler.

4. Multi-Stage Builds

The basic Dockerfile requires pre-building the JAR. A multi-stage build does everything inside Docker — no Java installation needed on the host:

Dockerfile (multi-stage)# Stage 1: Build
FROM eclipse-temurin:17-jdk AS build
WORKDIR /app
COPY pom.xml .
COPY .mvn .mvn
COPY mvnw .
RUN chmod +x mvnw
# Download dependencies first (cached layer)
RUN ./mvnw dependency:go-offline
COPY src ./src
RUN ./mvnw clean package -DskipTests

# Stage 2: Run (small image)
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
Why this matters

The final image only contains the JRE + your JAR. The JDK, Maven, and source code are discarded after the build stage. Result: ~200MB image instead of ~500MB+.

5. Docker Compose

Real applications need databases and other services. Docker Compose lets you define everything in one file:

docker-compose.ymlversion: '3.8'
services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://db:5432/taskdb
      - SPRING_DATASOURCE_USERNAME=admin
      - SPRING_DATASOURCE_PASSWORD=secret
    depends_on:
      - db

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: taskdb
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:

Start everything with one command:

Terminal# Start all services (build if needed)
docker compose up --build

# Run in background
docker compose up -d

# View logs
docker compose logs -f app

# Stop everything
docker compose down

6. Production Best Practices

Use .dockerignore

Prevent unnecessary files from bloating your image:

.dockerignore.git
.idea
*.md
target/
!target/*.jar
node_modules/

Set JVM Memory Limits

In containers, always tell the JVM about memory limits:

Dockerfile (ENTRYPOINT)ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]

Don't Run as Root

Dockerfile (security)# Create a non-root user
RUN addgroup --system appgroup && adduser --system appuser --ingroup appgroup
USER appuser

Health Checks

Dockerfile (health check)HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8080/actuator/health || exit 1
Quick checklist

✅ Multi-stage build   ✅ JRE not JDK   ✅ Non-root user   ✅ Memory limits   ✅ Health check   ✅ .dockerignore

7. Next Steps

  • Push to a registry — docker push to Docker Hub, AWS ECR, or GitHub Container Registry
  • Add CI/CD — Build images automatically with GitLab CI or GitHub Actions
  • Deploy to Kubernetes — Use Helm charts or plain YAML manifests
  • Monitor — Add Spring Boot Actuator + Prometheus + Grafana
  • Go native — Use GraalVM native images for sub-second startup
Docker doesn't just package your app — it standardizes your entire deployment pipeline. Learn it once, deploy anywhere.
